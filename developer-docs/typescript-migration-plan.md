# TypeScript Migration Plan

## Current State

The `online/src/worker/legacy/ts/` directory contains **434 TypeScript files** (~52K lines, excluding `core-java.ts`) generated by JSweet, organized as:

| Area | Files | Lines (est.) |
|------|------:|-----:|
| `com/vzome/core/` | 378 | ~45K |
| `com/vzome/desktop/` | 20 | ~3K |
| `com/vzome/fields/` | 4 | ~1K |
| `com/vzome/jsweet/` | 3 | ~1K |
| `com/vzome/{api,xml}` | 4 | ~500 |
| `java/` shims | 19 | ~1K |
| `org/w3c/dom/` | 5 | ~500 |

### Key characteristics of the generated code

1. **Pure namespace declarations** — every file wraps its content in `namespace com.vzome.core.algebra { ... }` with zero `import`/`export` statements. This assumes a single global scope stitched together by `tsc --outFile` or a `/// <reference>` graph.

2. **Dotted namespace names** — namespaces use Java's dotted package convention (`namespace com.vzome.core.algebra`), not nested `namespace com { namespace vzome { ... } }`. TypeScript supports dotted names, but they collapse into a single deeply-nested object tree at runtime.

3. **JSweet overloading pattern** — Java method overloads are encoded as separate methods with `$`-suffixed names (e.g., `createRational$long`, `createRational$long$long`) plus a dispatcher method that does runtime type-checking. Cross-references use bracket syntax: `n['plus$com_vzome_core_algebra_AlgebraicNumber'](...)`.

4. **Runtime interface checks** — Since TS interfaces are erased at runtime, JSweet attaches `MyClass["__class"]` and `MyClass["__interfaces"]` metadata, and `instanceof`-like checks use `constructor["__interfaces"].indexOf("com.vzome.core.algebra.AlgebraicNumber")`.

5. **j4ts dependency** — The `java.util.List`, `java.util.Map`, `java.util.Set`, `java.lang.Comparable`, etc. types come from the **j4ts** runtime library (`online/src/worker/legacy/candies/j4ts-2.1.0-SNAPSHOT/bundle.js`), which exports `java` and `javaemul` as global-style namespaces. The 19 `java/*.ts` files here are *additions* to what j4ts provides.

## The Core Problem

These files **cannot compile as ES modules** because:
- They have no `import`/`export` statements, so TypeScript treats them as **scripts** (global scope), not modules.
- If you add even one `import` or `export` to a file, TypeScript re-interprets it as a module, breaking all the cross-namespace references.
- The j4ts types (List, Map, etc.) aren't declared anywhere in `.d.ts` form — they only exist at runtime in the JS bundle.
- The `$`-suffix naming and bracket-access patterns are valid but opaque to TypeScript's type system.

## Recommendation: Incremental Module Migration with a Namespace-Preserving Barrel Strategy

### Strategy overview

Don't attempt a big-bang rewrite. Instead, create a **compilation scaffold** that lets the existing code compile today, then incrementally migrate files to proper ES modules while preserving the `com.vzome.*` namespace hierarchy as a module path convention.

### Phase 1: Make it compile as-is (namespace/script mode)

1. **Create a `tsconfig.json`** in `online/` with:
   ```json
   {
     "compilerOptions": {
       "target": "ES2022",
       "module": "None",
       "outFile": "src/worker/legacy/core-java-from-ts.js",
       "declaration": true,
       "strict": false,
       "skipLibCheck": true,
       "lib": ["ES2022"]
     },
     "include": ["src/worker/legacy/ts/**/*.ts"],
     "exclude": ["src/worker/legacy/ts/core-java.ts"]
   }
   ```
   With `"module": "None"` and `"outFile"`, TypeScript will concatenate all namespace files into a single JS output — matching how JSweet's JS output works today.

2. **Create an ambient `j4ts.d.ts`** declaring the `java.*` and `javaemul.*` types that come from j4ts at runtime. This is ~200 lines covering `java.util.{List, ArrayList, Map, HashMap, Set, HashSet, LinkedHashSet, Iterator, Collection, Collections, StringTokenizer}`, `java.lang.{Comparable, StringBuilder, System, IllegalArgumentException, ...}`, `java.io.*`, `java.math.BigInteger`, etc. You don't need full implementations — just interface/class declarations matching what the TS code references.

3. **Verify compilation** produces valid JS that's functionally equivalent to the current `core-java.js`.

### Phase 2: Convert namespaces to module paths (incremental)

This is the big win: each namespace like `com.vzome.core.algebra` maps naturally to a directory path that's *already the file system layout*. The migration for a single file looks like:

**Before** (`com/vzome/core/algebra/AlgebraicVector.ts`):
```typescript
namespace com.vzome.core.algebra {
  export class AlgebraicVector {
    field: com.vzome.core.algebra.AlgebraicField;
    // ...
  }
}
```

**After** (`com/vzome/core/algebra/AlgebraicVector.ts`):
```typescript
import type { AlgebraicField } from './AlgebraicField.js';
import type { AlgebraicNumber } from './AlgebraicNumber.js';

export class AlgebraicVector {
  field: AlgebraicField;
  // ...
}
```

Each directory gets a **barrel file** (`index.ts`) that re-exports everything, so existing consumers can import the whole "package" at once:
```typescript
// com/vzome/core/algebra/index.ts
export { AbstractAlgebraicField } from './AbstractAlgebraicField.js';
export { AlgebraicVector } from './AlgebraicVector.js';
export type { AlgebraicField } from './AlgebraicField.js';
export type { AlgebraicNumber } from './AlgebraicNumber.js';
// ...
```

This preserves the conceptual grouping of the namespace hierarchy. Files that haven't been migrated yet can coexist via a compatibility shim (see below).

### Phase 3: Replace JSweet-isms

As files are migrated to modules, clean up:

- **Overloaded methods**: Replace `$`-suffix methods with TypeScript overload signatures or union parameters. Example:
  ```typescript
  // Before (JSweet)
  createRational$long(n: number): AlgebraicNumber { ... }
  createRational$long$long(n: number, d: number): AlgebraicNumber { ... }
  createRational(n?: any, d?: any): any { /* dispatcher */ }
  
  // After (idiomatic TS)
  createRational(numerator: number, denominator?: number): AlgebraicNumber { ... }
  ```
  
- **Interface checks**: Replace `constructor["__interfaces"].indexOf(...)` with proper `instanceof` checks or TypeScript type guards.

- **`["__class"]` metadata**: Remove — TypeScript's type system handles this.

- **j4ts collections**: Replace `java.util.ArrayList<T>` → `Array<T>`, `java.util.HashMap<K,V>` → `Map<K,V>`, `java.util.HashSet<T>` → `Set<T>`, etc. This is the highest-value cleanup since it eliminates the j4ts runtime dependency entirely.

### Phase 4: Wire into the build

Once enough files are modules, switch esbuild to consume the `.ts` files directly (esbuild handles TS natively) instead of the concatenated JS. The `dynamic.js` entry point would import from the new module tree:
```typescript
import { com } from './ts/com/vzome/index.js';
// or, after full migration:
import { newDesign, loadDesign } from './ts/com/vzome/jsweet/index.js';
```

## Recommended Migration Order

Migrate **leaf dependencies first** (types with few cross-references), working inward:

1. **`java/` shims** (19 files) — Replace with native TS/JS equivalents or a thin `j4ts-types.d.ts`
2. **`org/w3c/dom/`** (5 files) — Replace with built-in DOM lib types
3. **`com/vzome/core/algebra/`** (26 files) — The foundation; interfaces + field implementations
4. **`com/vzome/core/construction/`** (45 files) — Depends mainly on algebra
5. **`com/vzome/core/math/`** — Symmetry, convex hull
6. **Other `core/` packages** — edits, commands, model, tools, exporters
7. **`com/vzome/desktop/`** — Controllers (may not be needed long-term)
8. **`com/vzome/jsweet/`** — The bridge layer (last, since it adapts the runtime)

## Why this approach over alternatives

| Alternative | Why not |
|---|---|
| **Rewrite from scratch in TS** | 52K lines of 20-year-old domain logic with exact arithmetic — too risky, too slow |
| **Keep as global scripts** | Works today (via JS), but blocks all TS tooling benefits: type-checking, IDE support, tree-shaking, incremental compilation |
| **Auto-convert all files at once** | The `$`-suffix overloading and `__interfaces` patterns require semantic understanding per-file — no reliable automated transform exists |
| **Use `declare namespace` wrappers** | Just adds a typing veneer without enabling module imports, tree-shaking, or eliminating j4ts |

The incremental barrel approach lets you **migrate one directory at a time**, keeping the build working at every step, while progressively gaining type safety and shedding the JSweet runtime overhead.
